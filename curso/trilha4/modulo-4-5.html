MÓDULO 4.5

🚀 Deploy no GitHub Pages

Publicando o curso TRIAD para o mundo — do repositório local ao site público com CI/CD automático e zero custo de hospedagem.

O que é

Para preparar o repositório: (1) na pasta raiz do projeto execute git init, (2) crie um .gitignore excluindo .env, node_modules e arquivos temporários, (3) git add . e git commit -m "initial commit", (4) crie o repositório no GitHub (público, sem README), (5) git remote add origin e git push -u origin main.

Por que aprender

O .gitignore correto é crítico — sem ele, você pode acidentalmente publicar tokens de API, configurações de modelos ou dados pessoais do soul.md. A preparação do repositório é a fundação de um deploy seguro.

Conceitos-chave

  • •.gitignore cobrindo .env e dados sensíveis
  • •Repositório público para GitHub Pages
  • •Branch main como fonte do Pages
  • •Commit semântico para histórico claro

O que é

No GitHub, acesse Settings → Pages → Source: Deploy from branch → Branch: main → folder: / (root). CRÍTICO: crie um arquivo vazio chamado .nojekyll na raiz do repositório. Sem ele, o GitHub Pages tenta processar o HTML com Jekyll e pode quebrar caminhos de arquivos com underscores ou diretórios especiais.

Por que aprender

O .nojekyll é o erro mais comum em deploys no GitHub Pages com HTML puro. Sem ele, arquivos em pastas começando com underscore ou underlines nos nomes podem não ser servidos corretamente.

Conceitos-chave

  • •.nojekyll na raiz — obrigatório para HTML puro
  • •Settings → Pages → Deploy from branch
  • •URL gerada: usuario.github.io/repo
  • •Deploy automático em cada push para main

O que é

GitHub Pages serve arquivos estáticos a partir da raiz do repositório. Todas as referências entre páginas devem usar caminhos relativos (href="curso/trilha1/index.html", não href="/curso/trilha1/index.html"). Links absolutos com barra inicial falham porque o repositório serve em /nome-do-repo/, não na raiz do domínio.

Por que aprender

Links absolutos quebram silenciosamente no Pages — a página carrega mas a navegação falha. Links relativos funcionam tanto localmente quanto no Pages sem nenhuma mudança, tornando o desenvolvimento e o deploy intercambiáveis.

Conceitos-chave

  • •Sempre usar caminhos relativos em href
  • •Nunca usar / no início de links internos
  • •../index.html para navegar para cima
  • •Teste local com Live Server antes do push

O que é

Com GitHub Actions, você cria um workflow em .github/workflows/deploy.yml que roda automaticamente em cada push para main: valida a estrutura HTML, verifica links quebrados com htmlproofer, e faz o deploy para Pages. Para projetos simples de HTML estático, o deploy automático via Settings → Pages já é suficiente — Actions é para quem quer validação antes do deploy.

Por que aprender

CI/CD automatiza a garantia de qualidade — você nunca publica acidentalmente HTML inválido ou links quebrados. Para um curso com dezenas de páginas, a validação automática é essencial para manter credibilidade.

Conceitos-chave

  • •Workflow em .github/workflows/deploy.yml
  • •Trigger em push para main
  • •htmlproofer para validar links
  • •Deploy bloqueado se validação falha

O que é

Tags git marcam pontos estáveis no histórico: git tag -a v1.0 -m "Trilha 1 completa" e git push origin v1.0. No GitHub, você pode criar Releases a partir de tags — com notas de versão, changelog e arquivos para download. Isso permite que alunos saibam exatamente quando o conteúdo foi atualizado e o que mudou.

Por que aprender

Releases dão transparência ao aluno — ele vê que a Trilha 4 foi adicionada na versão 4.0, que um módulo foi corrigido na 4.1. Isso aumenta confiança no material e facilita acompanhar as atualizações.

Conceitos-chave

  • •Tags anotadas para versões estáveis
  • •Releases com notas de mudanças
  • •Semantic versioning para o curso
  • •Changelog visível para alunos

O que é

Para equipes, o repositório precisa de: CONTRIBUTING.md com guia de como adicionar ou editar módulos (seguindo o design system), branch protection em main exigindo Pull Request aprovado antes de merge, e branch naming convention (feature/modulo-X-Y para novos módulos, fix/modulo-X-Y para correções). Isso garante que todos os colaboradores seguem o mesmo padrão.

Por que aprender

Sem governança, colaboradores introduzem inconsistências que desviam do design system. CONTRIBUTING.md e branch protection garantem qualidade independente de quem contribui.

Conceitos-chave

  • •CONTRIBUTING.md com guia do design system
  • •Branch protection exigindo PR aprovado
  • •Naming convention para branches
  • •Review obrigatório antes de merge em main

O que é

Para monitorar o curso no GitHub Pages, integre o Google Analytics (gratuito) adicionando o snippet gtag.js no head de todas as páginas. Acompanhe: pages mais visitadas, tempo médio por módulo, taxa de conclusão (via eventos de scroll depth), e origem do tráfego. Uma alternativa sem cookies é o Plausible Analytics — open-source e compatível com LGPD.

Por que aprender

Analytics transforma o curso de estático para melhorado por dados. Se o Módulo 1.3 tem abandono alto, talvez seja longo demais. Se o Módulo 4.1 recebe mais tráfego, talvez mereça mais conteúdo. Dados informam as próximas versões.

Conceitos-chave

  • •Google Analytics via gtag.js no head
  • •Eventos de scroll depth por módulo
  • •Plausible como alternativa sem cookies
  • •Dados informando evolução do conteúdo
