MÓDULO 1.5

🔀 OpenRouter

Um hub para todos os modelos — uma chave de API, centenas de modelos, roteamento inteligente e visibilidade de custo centralizada.

7
Tópicos
~35
Minutos
Básico
Nível
Setup
Tipo
O que é:

Usar múltiplos modelos de IA sem um hub significa: múltiplas chaves de API para gerenciar, múltiplos dashboards de custo para monitorar, múltiplos formatos de requisição diferentes, múltiplos rate limits para acompanhar e múltiplos pontos de falha para tratar. Cada modelo adicionado aumenta a complexidade operacional exponencialmente.

Por que aprender:

Entender o problema que o OpenRouter resolve justifica por que ele é um componente necessário do sistema, não apenas uma conveniência.

Conceitos-chave:
  • •Complexidade de múltiplas APIs
  • •Overhead operacional
  • •Proliferação de chaves
  • •Custo de manutenção
O que é:

O OpenRouter é uma API unificada que agrupa centenas de modelos de IA (Anthropic, OpenAI, Google, Meta, Mistral, DeepSeek e mais) sob uma única interface. Você cria uma chave de API no OpenRouter e acessa qualquer modelo usando o mesmo formato de requisição, sem mudar código, sem nova autenticação.

Por que aprender:

Uma chave para tudo é o princípio que viabiliza o Triad na prática. Sem esse hub, trocar de modelo no meio do fluxo exigiria mudança de código e de credenciais.

Conceitos-chave:
  • •API unificada
  • •Uma chave para N modelos
  • •Formato padrão de requisição
  • •Switching sem atrito
O que é:

O OpenRouter oferece um dashboard centralizado onde você pode ver uso e custo por modelo, por projeto e por período. É possível configurar alertas de custo, ver o histórico de chamadas e identificar quais modelos estão sendo mais utilizados. Isso é especialmente útil quando você tem múltiplos agentes rodando em paralelo.

Por que aprender:

Visibilidade de custo é o que permite otimizar um sistema de múltiplos modelos. Sem esse dado, você não sabe onde está gastando — e não pode decidir onde economizar.

Conceitos-chave:
  • •Dashboard unificado
  • •Custo por modelo
  • •Alertas de uso
  • •Visibilidade para otimização
O que é:

O OpenRouter suporta sufixos especiais nos nomes dos modelos que modificam o comportamento de roteamento. :nitro roteia para o provedor mais rápido naquele momento. :floor roteia para o provedor mais barato disponível. :auto deixa o OpenRouter escolher o melhor modelo para o seu prompt automaticamente, sem custo adicional.

Por que aprender:

Esses modificadores transformam o OpenRouter de um simples proxy em um sistema inteligente de roteamento. Para o Triad, usar :nitro no Condutor e :floor no Executor maximiza velocidade onde importa e economia onde é possível.

Conceitos-chave:
  • •:nitro para velocidade
  • •:floor para custo mínimo
  • •:auto para seleção inteligente
  • •Roteamento como otimização
O que é:

O OpenRouter permite que você conecte suas próprias chaves de API de provedores específicos. Para o DeepSeek V4, isso significa: criar conta na DeepSeek Platform, carregar créditos ($5-$10 inicialmente), gerar uma API key e conectá-la ao OpenRouter na seção "Bring Your Own Keys". A partir daí, chamadas ao DeepSeek usam sua chave diretamente, evitando rate limits do OpenRouter.

Por que aprender:

Para uso intensivo do Executor no Triad — especialmente overnight — rate limits podem interromper o trabalho. Ter chave própria da DeepSeek garante que o loop não seja interrompido.

Conceitos-chave:
  • •BYOK no OpenRouter
  • •DeepSeek Platform e créditos
  • •Rate limit próprio vs compartilhado
  • •Chave própria para uso intensivo
O que é:

O OpenRouter suporta configuração de fallbacks: se um modelo não responder dentro de X segundos ou retornar erro, a chamada é automaticamente roteada para um modelo alternativo que você definiu. Isso garante que o fluxo do Triad não pare por falha de um único provedor.

Por que aprender:

Em sistemas que rodam overnight ou em tarefas longas, falhas de modelos são inevitáveis. Fallbacks são o mecanismo de resiliência que mantém o trabalho continuando mesmo com indisponibilidades.

Conceitos-chave:
  • •Fallback automático
  • •Cadeia de modelos alternativos
  • •Continuidade de fluxo
  • •Tratamento de indisponibilidade
O que é:

O OpenRouter tem uma política de zero cobrança por respostas em branco ou com erro. Isso significa que se um modelo retornar uma resposta vazia, um timeout ou um erro de API, você não paga pelo token de output. Segundo o OpenRouter, essa política economiza ~$20.000/semana em toda a base de clientes combinada.

Por que aprender:

Em sistemas com muitas chamadas — especialmente no loop interno do Triad — chamadas com erro são inevitáveis. Não pagar por elas reduz o custo real de operação.

Conceitos-chave:
  • •Zero cobrança por erro
  • •Custo real vs custo estimado
  • •Impacto em loops de alta frequência
  • •Transparência de billing
